# Лабораторна "Вимірювання маси тіл"
## Demo Day — Технічний і педагогічний огляд

---

## 30-секундний пітч

Інтерактивна 3D лабораторна робота з фізики 7-8 класу. Учень виміряє масу трьох предметів (тенісний м'яч, яблуко, бейсбольний м'яч) трьома різними приладами (електронні ваги, важільні терези, динамометр) — **сам, без вчителя**. Система веде його через кожен крок: підказки в 3D просторі (стрілки, target rings), real-time показання приладів, автоматична перевірка результатів.

Працює в браузері на Promethean інтерактивних панелях у школах. Ніяких установок, акаунтів, інтернет-залежностей після першого завантаження.

---

## Чому ця лабораторна

### Методичний обґрунтування (для вчителя-методиста)

**Освітня тема:** "Вимірювання фізичних величин" (фізика 7 клас, перший розділ).

**Чому 3 прилади × 3 предмети:**
- **3 предмети різної маси** (58 г / 145 г / 180 г) — учень бачить різницю не тільки тактильно, а через інструменти.
- **3 принципово різні методи вимірювання:**
  - **Електронні ваги** — пряме автоматичне зчитування. Найшвидше, найпростіше.
  - **Важільні терези** — закон важеля, балансування гирями. Учень фізично переживає принцип рівноваги.
  - **Динамометр** — пружинний, вимірює СИЛУ (Ньютони), не масу. Учень має зрозуміти зв'язок F = m·g і конвертувати самостійно.
- **Перетин методів** дає розуміння: вимірювання — це не "просто зчитати число", а інтерпретація процесу.

**Чому інтерактивна 3D, а не просто відео:**
- Реальна фізика Rapier (open-source physics engine): балка терезів *насправді* нахиляється під дією маси, а не за анімацією. Учень може **експериментувати** — поставити дві гирьки на одну чашу, побачити що балка завалиться.
- Учень фізично *доторкається* до предметів через тач-панель Promethean. Найближче що можна зробити до реальних рук-в-лабораторії, без витрат на справжні комплекти.

### Демонстраційні моменти

Що показувати в порядку важливості:

1. **Guided mode reveal** (1 хв) — натиснути "Почати", показати як стрілка вказує на тенісний м'яч → перетягнути → стрілка одразу переходить на електронні ваги → дисплей показує 58 г → введення значення → автоматичний перехід до наступного завдання.
   > Ключовий момент: "Учень не потребує вчителя поряд. Підказка завжди з ним. Помилитися важко."

2. **Важільні терези — справжня фізика** (1 хв) — покласти яблуко на ліву чашу. Балка нахиляється. Підказка адаптивно показує "Додай ще ~80 г на праву". Покласти 100 г — балка майже вирівняна. Замінити на 50+20+10 — точний баланс. Підказка: "✓ Балка вирівняна. Маса = 180 г."
   > Ключовий момент: "Це не скрипт. Це Rapier physics engine. Учень може ставити гирьки в довільному порядку, на будь-яку чашу — все працює."

3. **Динамометр — методична глибина** (30 сек) — підвісити м'яч на гачок. Пружина візуально розтягується. Шкала показує 0.57 N. Підказка: "Прочитай Ньютони і впиши". Журнал автоматично показує перерахунок у грамах.
   > Ключовий момент: "Тут не просто вимірювання — тут учень розуміє різницю між масою і вагою."

4. **Repository walkthrough** (1 хв) — відкрити VSCode/Finder, показати:
   - `docs/superpowers/specs/` — два дизайн-документи (v1 + v2)
   - `docs/superpowers/plans/` — три плани реалізації (детальні, з кодом)
   - `git log --oneline` — 60+ комітів, чітка історія, "fix → revert → re-plan" як приклад інженерної дисципліни
   - `npm run test` — 25 тестів, всі зелені
   > Ключовий момент: "Це не demo-prototype. Це продукт з перевіреною архітектурою."

5. **Skip guidance toggle** (10 сек) — натиснути "🎓 Гід: Вимк". Стрілки зникають. Залишається базовий HUD з показаннями. Knопка персистентна (localStorage).
   > Ключовий момент: "Просунутий учень або вчитель-демонстратор може вимкнути підказки одним кліком."

## Технічна якість (для розробника)

### Стек

- **React 19 + TypeScript 6 + Vite 8** — сучасний frontend stack, hot reload, strict mode типчек
- **Three.js / React Three Fiber 9** — декларативна 3D в JSX
- **Rapier 2** (через `@react-three/rapier`) — WASM physics engine, той самий що в Bevy game engine
- **Zustand 5** — мінімалістичний state manager без boilerplate Redux
- **Vitest 4** — швидке тестування з jsdom для smoke-тестів компонентів

### Архітектура — декомпозиція по domains

```
src/
  utils/        ← pure functions (units, tolerance) — 100% test coverable
  lab/          ← lab logic: tasks, state machine, journal, screens
  scene/        ← 3D scene: lighting, camera, table, instruments, objects
  physics/      ← drag system, snap targets registry
  guided/       ← guided mode: TaskSteps, StepEngine, overlays, primitives
  ui/           ← reusable UI primitives (Button, GlassPanel, NumberInput, Keypad)
```

**Кожна папка — одна відповідальність.** Можна розуміти і змінювати незалежно. Файлів багато — кожен невеликий і фокусований.

### Декларативні TaskSteps — масштабованість

Кожне з 9 завдань описане як декларативний step tree:

```ts
{
  id: 'pickup',
  target: { kind: 'object', id: 'tennis-ball' },
  visualHint: 'arrow',
  hintTemplate: 'Натисни і потримай тенісний м\'яч',
  complete: { kind: 'dragging', bodyPattern: 'tennis-ball' },
}
```

Auto-detection через існуючі stores (useReadings, useDrag, snap targets) — нуль ручного state management.

### Тестова disciplina

- **25 unit-тестів** (vitest) — pure logic покрита: utils/units, lab/tasks, lab/LabState, guided/TaskSteps, guided/StepEngine
- TDD підхід: тести написані ПЕРЕД реалізацією для всіх pure modules
- React Three Fiber компоненти не тестуються unit-тестами (потрібен реальний WebGL) — manual QA в браузері

### Перформанс

- **Бандл: ~3.3 МБ** (Three.js + Rapier WASM = більшість)
- **DPR обмежено [1, 1.5]** — на high-DPI дисплеях рендер 2× менше пікселів
- **Контактні пари приладів throttle 1/6 frames** — сума мас читається 10 разів/сек, не 60
- **Немає shadows / HDRI / post-processing** — стабільні 60 fps на середньому ноутбуці без дискретної GPU
- **Bezpieczna physics при низькому FPS** — fixed timestep 1/60, без tunneling

### Дисципліна процесу — git history як артефакт

```
docs(plans): v2-lite — 22 tasks across 4 phases, no GPU-heavy changes
revert: roll back v2 Phase 1 entirely — return to last-known-good v1 state
perf: aggressive FPS fix — disable shadows, remove HDRI, ...
docs(specs): v2 design — Apple Studio + persistent guided mode
docs(plans): v2 implementation plan — 31 tasks across 6 phases
docs(specs): v1 design — Mass Measurement Lab
```

Це показує: **спочатку дизайн → план → реалізація. Коли виявилися проблеми (perf v2 Phase 1) — чесний revert + переплановування → v2-lite план → виконання за планом.** Без героїчних спасінь, без "просто пофіксимо в production".

## Що людина запитає (FAQ для пітчу)

**Q: Чому не використовуєте PhET (готові симуляції MIT)?**
A: PhET — Java/HTML5 але старий і не оптимізований під тач. Ця лабораторна — нативно для тач-панелей (нетривіальні snap-targets, gesture-based drag), Apple HIG aesthetics для сучасного відчуття. PhET красиво для алгоритмів, ця — для досвіду.

**Q: А інші лабораторні?**
A: Зараз фокусуємось на тому, щоб ця ОДНА була зразковою. Архітектура (декларативні TaskSteps) дозволяє додавати нові, але це окремий продуктовий цикл — не паралельно.

**Q: Як це продаватиметься школам?**
A: (Відповідь — це питання до інвестора, поки мовчати або сказати "обговорюваний бізнес-аспект, технічно платформа готова до beta-тестування у школах").

**Q: Скільки часу на розробку?**
A: ~5 робочих днів повного циклу: 2 дні на v1 MVP (брейнсторм → план → код), 1 день v2 спроба + revert, 2 дні v2-lite + Phase A/B/C полірування. **Все задокументовано і відтворювано.**

**Q: Тести — чому unit-тести а не E2E?**
A: E2E тести 3D drag-and-drop не виконуються надійно (потрібен Playwright з WebGL emulation, що крихко). Unit-тести покривають 100% бізнес-логіки (math, state machine, step engine). Manual QA на реальному пристрої — для візуального і тактильного.

## 1. Цілі та межі MVP

**Що ми робимо:**
Браузерну 3D-симуляцію однієї лабораторної роботи "Вимірювання маси тіл" для учнів 6-7 класу. Учень визначає масу трьох предметів (тенісний м'яч, яблуко, бейсбольний м'яч) трьома різними приладами (динамометр, важільні терези, електронні ваги), вводить результати в цифровий лабжурнал і отримує фінальний звіт з порівнянням з еталоном.

**Що ми НЕ робимо в MVP** (свідомо):
- Авторську систему / редактор лабораторних — лаба hard-coded.
- Систему акаунтів, прогресу, LMS-інтеграцій — без бекенду.
- Інші лабораторні — фокус на одній.
- Підтримку iPad/Safari/Firefox — тільки Chrome на Promethean.
- Похибки вимірювання — прилади показують точні значення.
- Складну ергономіку (правильний кут утримання тощо).

**Цільовий пристрій:** інтерактивна панель Promethean ActivPanel (Android або Windows OPS, Chrome 120+), 4K, мульти-тач до 20 точок.

## 2. Ключові архітектурні рішення

## 3. Архітектура

### Структура модулів

```
src/
  App.tsx                    головний компонент: <Canvas> + UI overlay
  scene/
    Scene.tsx                <Canvas>: освітлення, камера, Physics
    Table.tsx                модель столу
    instruments/
      Dynamometer.tsx
      LeverBalance.tsx
      DigitalScale.tsx
    objects/
      TennisBall.tsx
      Apple.tsx
      Baseball.tsx
      Weights.tsx            гирьки 1кг, 500г, 200г, 100г, 50г, 20г, 10г
  physics/
    PhysicsWorld.tsx         <Physics> wrapper з налаштуваннями
    DragController.tsx       тач/миша → kinematic translation
  lab/
    tasks.ts                 список 9 завдань (TS-константа)
    LabJournal.tsx           таблиця результатів
    TaskPanel.tsx            панель поточного завдання
    SummaryScreen.tsx        фінальний звіт
    IntroScreen.tsx          стартовий екран
  store.ts                   Zustand: phase, currentTaskIndex, journal
  assets/                    .glb моделі, HDRI (статично)
```

### Стан додатку (Zustand)

```ts
type LabPhase = 'intro' | 'in-progress' | 'finished'

type LabState = {
  phase: LabPhase
  currentTaskIndex: number
  journal: { taskId: string; userValue: number; timestamp: number }[]
  start(): void
  setMeasurement(taskId: string, value: number): void
  nextTask(): void
  finish(): void
  reset(): void
}
```

### Потік даних

1. Учень тягне об'єкт пальцем → `DragController` застосовує `setNextKinematicTranslation` до RigidBody.
2. Об'єкт лежить на приладі → прилад читає масу з фізичного світу (через колізії або суму контактуючих тіл).
3. Прилад оновлює власну візуалізацію (canvas-текстура LCD, поворот стрілки, розтяг пружини).
4. Учень читає значення в 3D, вводить його в input bar внизу.
5. На "Записати → Далі" значення йде в Zustand → перехід до наступного завдання.
6. Після 9-го запису — `phase = 'finished'`, рендериться `<SummaryScreen>`.

## 4. 3D-сцена та ассети

### Розкладка столу

Стіл 2.5 м × 1.2 м у просторі сцени (1 одиниця Three.js = 1 метр). Зони на стільниці:
- **Ліва частина** — рамка з трьома об'єктами (тенісний м'яч, яблуко, бейсбольний м'яч).
- **Центр-ліво** — динамометр (висить вертикально на штативі).
- **Центр** — важільні терези.
- **Центр-право** — електронні ваги.
- **Праворуч знизу** — набір гирьок (1 кг, 500 г, 200 г, 100 г, 50 г, 20 г, 10 г).

### Камера

`<PerspectiveCamera>` фіксована, FOV 50°, позиція (0, 1.5, 2.0), дивиться на (0, 0.85, 0). Без `OrbitControls`.

Опційні кнопки в UI: "Наблизити до приладу" (анімований lerp до пресету для кожного приладу) + "Скинути" (повернутися до базового ракурсу).

### Освітлення

3-точкова схема: key light (теплий 4500K, стельова лампа), fill light (м'яке заповнення фронту), rim light (контурний ззаду). `softShadows` тільки під ключовими об'єктами. Environment HDRI з drei (preset "warehouse" або кастомний "клас").

### Стратегія ассетів

**Безкоштовні джерела:** Sketchfab CC0/CC-BY, Poly Haven (CC0), Free3D.

**AI-генерація** (Meshy.ai / Tripo3D / Genie) для того, що не знайдемо безкоштовно — переважно прості форми (фрукти, м'ячі) працюють добре.

**Список ассетів (~15 окремих моделей):**
- Дерев'яний лабораторний стіл (1)
- Динамометр з пружиною (5N) + штатив (2)
- Важільні терези з двома чашами і стрілкою (1)
- Електронні ваги (1)
- 7 гирьок різних номіналів (7)
- Тенісний м'яч, яблуко, бейсбольний м'яч (3)

**Формат:** glTF/GLB через `useGLTF`. Завантаження — `preload: true` для критичних, lazy — для другорядних. Усі моделі нормалізуються до метричного масштабу через `<Asset>`-обгортку.

**Розмір бандлу:** 50–100 МБ моделей + 5–10 МБ HDRI. Хостимо на Cloudflare Pages (CDN, 25 МБ ліміт на файл — оптимізуємо великі моделі через gltf-transform).

**Ризик:** AI-генеровані моделі поки слабкі на дрібних деталях шкал/циферблатів. Якщо це стане критичним — стиль може зсунутися від "фотореалістичного" до "стилізованого реалізму".

## 5. Фізика та взаємодія

### Налаштування фізичного світу

```tsx
<Physics gravity={[0, -9.81, 0]} timeStep="vary">
  <Table />          // fixed
  <Instruments />    // mix of fixed bodies + revolute joints (важелі)
  <DraggableObjects /> // dynamic
</Physics>
```

### Маси (реалістичні)

- Тенісний м'яч — 58 г
- Яблуко — 180 г
- Бейсбольний м'яч — 145 г
- Гирьки — точно за номіналом

### Drag-and-drop

1. **pointerdown** на об'єкті → знаходимо найближчий `RigidBody` під курсором (raycast), фіксуємо його `pointerId`.
2. Перемикаємо тіло в `kinematic` режим, заморожуємо `angularVelocity`.
3. **pointermove** → оновлюємо `setNextKinematicTranslation` так, щоб об'єкт слідував за пальцем по площині на висоті ~10 см над столом.
4. **pointerup** → повертаємо в `dynamic`, об'єкт падає під гравітацією.
5. Exponential smoothing на позицію (Promethean інколи реєструє "тремтіння" пальця).
6. Обмеження межами столу — об'єкт не може виходити за bbox столу.

**Мульти-тач:** кожен `pointerId` тримає свій активний RigidBody. Двоє учнів можуть одночасно тягати різні предмети.

**Snap до гачка динамометра:** при дропі в радіусі 5 см від гачка — об'єкт фіксується через `FixedJoint` з невеликою анімацією.

## 6. Поведінка приладів

### Електронні ваги

Платформа — `<RigidBody type="fixed">` з `IntersectionEvents`. Кожен кадр сумуємо маси всіх контактуючих тіл, виводимо на LCD з точністю 1 г. LCD реалізовано як `<mesh>` з canvas-текстурою (256×128 px, семисегментний шрифт).

Кнопка "Tare" обнуляє показання (запам'ятовує поточне як zero offset).

### Важільні терези — справжній фізичний симулятор

Конструкція: `Stand (fixed) → Beam (dynamic, RevoluteJoint у центрі) → Left pan + Right pan (FixedJoint до beam)`.

Учень кладе об'єкт на ліву чашу → балка нахиляється вліво (Rapier фізика). Учень додає гирьки на праву, поки балка не вирівняється. **Маса об'єкта = сума мас гирьок на правій чаші.**

Стрілка приварена до beam. Жодного "автоматичного значення" — учень повинен сам зрозуміти.

Це найдидактичніший момент усієї лабораторної.

### Динамометр

Конструкція: `Stand → Spring top (fixed) → Spring (visual mesh, не фізичне тіло) → Hook (kinematic, рухається вертикально)`.

Учень чіпляє об'єкт на гачок (snap при наближенні). Сила натягу `F = m × g`. Hook опускається пропорційно `Δy = F / k` (k = 50 N/m, діапазон 0-5N). Пружина візуально розтягується.

Учень читає значення в **Ньютонах** (бо це сила!) і вводить його в інпут. Журнал автоматично показує поруч перерахунок у грами (`m = F / 9.8`). Освітній момент зберігається — учень бачить обидва значення і розуміє звʼязок, — але без двокрокового вводу. Перевірка `expectedValue ± tolerance` йде по полю в Ньютонах (`inputUnit: 'N'`).

## 7. UI та потік лабораторної

### Структура екрану

- **Header (зверху)** — назва лабораторної + прогрес з 9 капсул + лічильник "X/9".
- **Task panel (зліва, ~26% ширини)** — поточне завдання + коротка підказка.
- **Lab journal (справа, ~26% ширини)** — таблиця 9 записів. Записані — синім; поточний — затемнений з прочерком; майбутні — тьмяні.
- **Input bar (знизу по центру)** — числове поле + кнопка "Записати → Далі". Нативна тач-клавіатура браузера.
- **Camera buttons (знизу справа)** — "Наблизити" + "Скинути".

### Структура завдань (`lab/tasks.ts`)

```ts
type Task = {
  id: string
  objectId: 'tennis-ball' | 'apple' | 'baseball'
  instrumentId: 'digital-scale' | 'lever-balance' | 'dynamometer'
  prompt: string         // "Виміряйте масу яблука важільними терезами."
  hint: string
  expectedValue: number  // у грамах (або Ньютонах для динамометра)
  tolerance: number      // частка, наприклад 0.10 = ±10%
  inputUnit: 'g' | 'N'
}
```

9 завдань: 3 предмети × 3 прилади. Порядок — спочатку всі вимірювання тенісним м'ячем, потім яблуком, потім бейсболом.

### Стан-машина

```
intro → (Почати) → in-progress (taskIndex 0..8) → (9-й запис) → finished → (Почати знову) → intro
```

### Перевірка значень

При натисканні "Записати": порівнюємо `userValue` з `expectedValue ± tolerance`. У журналі запис фарбується:
- 🟢 в межах допуску
- 🟡 поза допуском, але близько (×1.5 tolerance)
- 🔴 далеко

**Не блокує перехід далі** — учень бачить підсумок у фіналі. Педагогічно правильно: вчитель може потім розібрати помилки.

### Підсвічування активного приладу

Прилад поточного завдання отримує золотисту обводку (Outline постпроцес від drei). Кутом ока зрозуміло, куди йти.

### Фінальний екран

Модальне вікно з:
- Повним журналом усіх 9 значень.
- Кольоровими галочками (🟢/🟡/🔴) поряд з кожним.
- Підсумком "X з 9 точно".
- Кнопкою "Зробити screenshot" (через `html-to-image`).
- Кнопкою "Почати знову" (`reset()` стану).

Без оцінки в балах — лише візуальний фідбек.

**Бюджет на 60 FPS:**
- ~20 dynamic RigidBody одночасно.

## 9. Відкриті питання / ризики

1. **Якість AI-генерованих моделей приладів.** Ризик низькоякісних шкал/циферблатів. **Mitigation:** перевіряємо рано, на етапі заготовки приладів. Якщо погано — зсуваємося в стилізований реалізм (немає необхідності в дрібних реалістичних деталях).

2. **Тач-чутливість Promethean.** Реальні панелі можуть давати специфічні артефакти (ghost touches, лаги при мульти-тачі). **Mitigation:** smoothing на позицію + рання перевірка на реальному пристрої.

3. **Розмір ассетів.** Якщо HDRI + моделі перевищать ~150 МБ — перший лоад буде довгим. **Mitigation:** агресивний gltf-transform compression, відкладене завантаження неактивних об'єктів.

4. **Інтернет у школах.** Питання деплою вирішено на користь хмари (варіант A), але якщо в реальних класах інтернет нестабільний — повертаємося до PWA-варіанту (відносно безболісний міграційний шлях у Vite).

## 10. Що далі

Після затвердження цього дизайну — переходимо до плану імплементації (`writing-plans` skill), який розбиває роботу на конкретні етапи з оцінками часу.

Орієнтовні етапи (для контексту):
1. Setup проекту (Vite + R3F + Rapier + TypeScript). ~півдня.
2. Базова сцена — стіл, освітлення, камера. ~1 день.
3. Drag-and-drop без приладів (просто кидаємо м'яч на стіл). ~1-2 дні.
4. Електронні ваги (найпростіший прилад). ~1 день.
5. Динамометр. ~1-2 дні.
6. Важільні терези (найскладніший). ~2-3 дні.
7. Лабжурнал, завдання, фінальний екран. ~2-3 дні.
8. Підбір/генерація фінальних ассетів. ~2-3 дні.
9. Деплой і тести на реальній панелі. ~1-2 дні.

**Сумарно:** ~3-4 тижні роботи одного розробника при повному фокусі.
